# /implement - 实施执行阶段

执行端到端交付流程的第四阶段：实施执行。

**前置条件**: 必须先完成 Design 阶段

## 使用方式

```bash
# 启动实施执行阶段
/implement

# 执行特定任务
/implement "task-1"

# 从中断处继续
/implement --resume
```

## 阶段目标

按照 TDD 原则实施功能，确保代码质量。

## TDD 工作流程

```
┌─────────┐    ┌─────────┐    ┌─────────┐
│   RED   │───▶│  GREEN  │───▶│ REFACTOR│
│ 编写测试 │    │ 实现代码 │    │  重构  │
└─────────┘    └─────────┘    └─────────┘
```

### RED - 编写失败的测试
1. 定义接口/类型
2. 编写测试用例
3. 运行测试确认失败

### GREEN - 实现最小代码
1. 实现功能
2. 运行测试确认通过

### REFACTOR - 重构改进
1. 在测试保护下重构
2. 确认测试仍然通过

## 审查流程

### 两阶段审查
```
实施 → 自审 → 规格审查 → 修复 → 质量审查 → 修复 → 完成
```

### 1. 自审
- 功能完整性检查
- 代码质量检查
- 测试质量检查
- 安全检查

### 2. 规格审查
- 确认实现符合规格
- 检查无遗漏功能
- 检查无额外功能

### 3. 质量审查
- 代码简洁性
- DRY 原则
- 设计模式
- 项目规范

## 质量标准

### 代码质量
- 函数长度 ≤ 50 行
- 文件长度 ≤ 800 行
- 嵌套深度 ≤ 4 层
- 命名清晰有意义

### 测试质量
- 测试覆盖率 ≥ 80%
- 快乐路径覆盖
- 边界条件覆盖
- 错误场景覆盖

### 提交规范
```bash
git commit -m "feat: add user authentication

- Implement login functionality
- Add JWT token validation
- Handle error cases

Tests: 5/5 passing
Coverage: 95%"
```

## 质量门禁

完成实施执行前，确认:
- [ ] 严格遵循 TDD 流程
- [ ] 测试覆盖率 ≥ 80%
- [ ] 所有测试通过
- [ ] 代码符合规范
- [ ] 自审完成
- [ ] 审查问题已修复

## 下一阶段

实施执行完成后，进入质量验证阶段:
```bash
/phase:verify
```
