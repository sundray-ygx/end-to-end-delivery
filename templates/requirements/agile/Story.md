# User Story 模板

交付周期：3~5 天（完成开发测试）

User Story（用户故事）是一个具体的用户操作场景，从用户视角描述用户所希望获得的功能。

**示例：** 作为管理员，我想要创建管理员账号，以便于交付给公司管理员使用。

User Story 需要满足 **INVEST 原则**，故事细化过程中会补充业务细节和验收条件，故事是工作量估算的基本单位，也是最小可测试单元（开发周期包含测试）。一个 User Story 会被分解为一个或多个 Task（具体的工作项）。

```
Feature ──► User Story ──► Tasks
```

---

## INVEST 原则

| 原则 | 说明 |
|-----|------|
| **Independent（独立）** | 用户故事必须彼此独立，低耦合 |
| **Negotiable（可协商）** | 用来提醒团队和干系人要进行讨论，而不是直接作为契约使用 |
| **Valuable（有价值）** | 对用户或客户来讲必须是重要的，有价值的 |
| **Estimable（可估算）** | 开发团队必须能够估算创建用户故事所需的工作量 |
| **Small（规模小且适中）** | 尽可能要在一个迭代内完成（建议 3~5 个工作日） |
| **Testable（可验证）** | 用户故事必须是可以被验证的 |

> 在现实工作中，如无法同时满足 6 个原则，至少要满足：**EST > INV**（可估算、规模小且可验证）

## 用户故事模板

### 用户故事

**责任人：** PO

**用户视角：**

```
作为 <用户角色>，
我想要 <完成 xxx 活动>，
以便于 <实现 xxx 价值>
```

### A/C 验收条件

**责任人：** PO & TL & TSE

#### 一、功能性验收条件

**主：PO，副：TL & TSE**

1. **业务规则**：满足哪些约束条件（含功能、系统、UI...），提供交互稿 & 视觉稿
2. **实例化场景描述**（功能使用时的正常流程和异常流程）：
   ```
   Given（前提条件）- When（行为）- Then（结果）
   ```

#### 二、非功能性验收条件

**主：TL & TSE，副：PO**

例如：性能要求、稳定性要求、安全要求等等

#### 三、DFX、可测试性

**主：TSE，副：TL**

### 依赖与风险

**责任人：** PO & TL & TSE

| 类型 | 描述 |
|-----|------|
| **需求关联性** | 内部功能依赖，外部跨团队协作 |
| **技术风险** | 技术风险、测试风险 |

### 技术思路

**责任人：** TL & 模块负责人

技术如何实现，技术方案概况。
